# Project Analysis: Godot Plush Character

## Overview
Multiplayer action RPG built with Godot 4.6, featuring state machine-based character animations, inventory, quests, spells, and combat systems.

## Tech Stack

### Core
- **Engine**: Godot 4.6 (GL Compatibility renderer)
- **Language**: GDScript
- **Multiplayer**: WebSocket-based (WebSocketMultiplayerPeer), server-authoritative
- **Port**: 25565, max 4095 clients

### Key Addons
1. **godot_state_charts**: Finite state machine for character states (Ground, Climbing, Water, Knockback)
2. **terrain_3d**: Terrain generation and rendering
3. **proton_scatter**: Particle/vegetation scattering system
4. **mixamo_animation_retargeter**: Animation retargeting for Mixamo animations
5. **LineRenderer**: Visual effects/lines
6. **script-ide**: Type checking and IDE support

## Architectural Patterns

### 1. Component-Based Architecture
```
Player (CharacterBody3D)
├── StateChart (StateMachine)
│   ├── GroundState
│   ├── ClimbingState
│   ├── WaterState
│   └── KnockbackState
├── CombatComponent (Node3D)
├── SpellManager (Node3D)
├── Inventory (Node3D)
├── VisualRoot (Node3D)
│   └── Dude (MeshInstance3D)
└── OrbitView (Camera3D)
```

### 2. State Machine Pattern
- **Implementation**: Uses `godot_state_charts` addon
- **States**: Ground, Climbing, Water, Knockback
- **Transitions**: Event-driven via `state_chart.send_event("event_name")`
- **Separation**: Each state handles its own physics and inputs

### 3. Multiplayer Architecture
- **Pattern**: Server-authoritative with RPC synchronization
- **Key RPCs**:
  - `@rpc("authority", "reliable", "call_local")` - Server commands
  - `@rpc("any_peer", "reliable", "call_local")` - Client requests
  - `@rpc("any_peer", "call_remote", "unreliable_ordered")` - State sync
- **Sync Strategy**: 
  - Server broadcasts state via `server_state.rpc()`
  - Clients interpolate via `create_tween()` (0.1s lag compensation)
  - `rpc_id()` used for peer-specific communication

### 4. Resource-Based Data System
- **SpellResource**: Spells defined as `.tres` resources with enums for types
- **QuestResource**: Quest data as resources
- **ItemResource**: Inventory items with sell prices and actions
- **Materials/Shaders**: `.tres` for reusable visual data

### 5. Game Loop Structure
- **MM (MultiplayerManager)**: Autoload with tick/slow_tick signals
- **Tick**: Main game loop for updates
- **Slow Tick**: Combat stat calculations (HP/MP regen, threat tables)

## Code Style & Conventions

### File Naming
- **Scripts**: `snake_case.gd` (e.g., `player_combat.gd`, `spell_manager.gd`)
- **Scenes**: `snake_case.tscn` or `PascalCase.tscn` (mixed usage)
- **Resources**: `snake_case.tres`

### Variable/Class Naming
- **Classes**: `PascalCase` with `class_name` (e.g., `class_name Player`, `class_name CombatComponent`)
- **Variables**: `snake_case` (e.g., `jump_height`, `spell_index`, `current_state`)
- **Constants**: `UPPER_SNAKE_CASE` (e.g., `MAX_SLOTS`, `JUMP_VELOCITY`)
- **Signals**: `snake_case` (e.g., `died`, `health_changed`, `tick`)
- **Nodes in Scenes**: `PascalCase` with `%` shortcuts (e.g., `%VisualRoot`, `%HealthBar`)

### Type Hints
- **Godot 4 Native Types**: `int`, `float`, `String`, `bool`, `Vector2`, `Vector3`, `Node`, `Node3D`
- **Function Signatures**: Always typed (e.g., `func _ready() -> void:`, `func transition_to(state: ANIMATION_STATE):`)
- **Arrays**: `Array[Player]`, `Dictionary`

### Decorators & Annotations
- `@export`: Inspector-exposed variables
- `@onready`: Cached node references (performance optimization)
- `@rpc(...)`: Multiplayer RPCs with parameters:
  - `"authority"` (server only) or `"any_peer"`
  - `"reliable"` or `"unreliable_ordered"`
  - `"call_local"`, `"call_remote"`, or `"call_remote_and_local"`

## Key Systems

### Combat System (`target_dummy/combat_component.gd`)
- **Stats**: strength, agility, intelligence, stamina, mp5, hp5
- **Threat Table**: Dictionary tracking aggro per player ID
- **Damage Queue**: `changes_received` dictionary buffers damage per tick
- **Leveling**: EXP-based with 1.23x scaling per level

### Spell System (`spellManager/`)
- **SpellManager**: Handles spell casting and UI
- **Spell Types**: Enums for PROJECTILE, INSTANT, etc.
- **Projectile Types**: FIREBALL, FROSTBOLT, etc.
- **Resource-based**: Spells loaded as `.tres` resources

### Inventory System (`components/inventory/`)
- **Grid-based**: 16 slots max
- **ItemResource**: Base resource for items with sell prices and actions
- **UI**: CanvasLayer with GridContainer

### Quest System (`questManager/`)
- **QuestManager**: Autoload handling quest UI
- **QuestResource**: Base resource with name, description, goal text
- **Types**: Search quests, target dummy quests, chains

### Achievement System (`achievement/`)
- **Autoload**: `Achievements` scene
- **Triggers**: Coin collection, etc.

## Multiplayer Patterns

### Authority Checks
```gdscript
if multiplayer.is_server():
    # Server logic
if multiplayer.get_unique_id() == id:
    # Local player logic
```

### State Synchronization
```gdscript
# Server broadcasts
@rpc("authority", "reliable")
func server_state(new_position: Vector3, new_rotation: Vector3):
    # Interpolate remote players

# Client sends input
@rpc("any_peer", "call_remote", "unreliable_ordered")
func client_state(new_position: Vector3, new_rotation: Vector3):
    server_state.rpc(new_position, new_rotation)
```

### Node Cleanup
- Remote players remove state machines: `$StateChart.queue_free()`
- Auto-attack disabled for remote players
- UI removed for non-local players

## Physics & Movement

### Movement States
- **Ground**: WASD + mouse look, coyote jumping, squash/stretch
- **Climbing**: Attach to objects, vertical movement only
- **Water**: Swimming physics
- **Knockback**: Forced movement with decay

### Physics Settings
- Physics interpolation enabled
- `move_and_slide()` for movement
- Custom gravity: jump_gravity (0.4s to peak), fall_gravity (0.3s descent)

## Visual Effects
- **Particles**: Jump particles, land particles, movement dust
- **Tweening**: `create_tween()` for smooth interpolation (0.1s lag)
- **Squash & Stretch**: Animation feedback on jumps/landings
- **Shaders**: Custom shaders for spells (fireball, frostbolt)

## Notable Design Decisions

1. **Separation of Concerns**: Combat, spells, inventory, and movement are separate nodes
2. **Event-Driven States**: State transitions via signal-like events
3. **Server Tick**: Centralized tick system for combat calculations
4. **Resource Reusability**: Heavy use of `.tres` for data-driven design
5. **RPC Granularity**: Specific `rpc_id()` calls for peer-specific updates

# Developer Onboarding & Coding Standards Guide

## Project Architecture

### Data Flow Overview

This is a **server-authoritative multiplayer game** with the following data flow:

```
┌─────────────────────────────────────────────────────────────┐
│                      CLIENT (Player)                        │
├─────────────────────────────────────────────────────────────┤
│  Input → State Machine → Physics → RPC → WebSocket → Server │
│  ← RPC Sync ← ← State Interpolation ← ← Player Updates      │
└─────────────────────────────────────────────────────────────┘
```

**1. Client-Side (Local Player)**
- **Input Layer**: `_unhandled_input()` captures player input (WASD, mouse, jump, attack)
- **State Machine**: `StateChart` routes to GroundState/ClimbingState/WaterState/KnockbackState
- **Physics**: `move_and_slide()` updates position, velocity calculated per-state
- **RPC Layer**: `send_state()` broadcasts position/rotation to server every tick

**2. Server-Side (Authority)**
- **MultiplayerManager (Autoload)**: Hosts WebSocket server (port 25565), manages client connections
- **Combat Component**: Processes damage via `slow_tick` signal (throttle mechanism)
- **State Validation**: Server validates inputs, calculates combat results, broadcasts via `server_state.rpc()`

**3. Remote Clients**
- **Interpolation**: `create_tween()` smooths remote player movement (0.1s lag compensation)
- **Visual Sync**: `server_state()` RPC updates position/rotation of other players

**4. Data Persistence**
- Currently **no database** - session-based multiplayer
- Player data stored in `Dictionary` (position, stats) passed via `add_client(id, data)`
- Resources (.tres) loaded from disk at runtime for spells/items/quests

### Key Architectural Patterns

**Component Composition**
```
gdscript
Player (CharacterBody3D)
├── StateChart (StateMachine)
├── CombatComponent (stats, health, threat)
├── SpellManager (casting, projectiles)
├── Inventory (16 slots, ItemResource)
└── VisualRoot (mesh, animations, VFX)
```

**Resource-Based Data**
- Spells, Items, Quests defined as `.tres` resources
- Data-driven design: modify `.tres` files to balance game without code changes
- `class_name` used for type safety: `class_name SpellResource`, `class_name ItemResource`

**Tick-Based Systems**
- `MM.tick` (fast): Movement, physics, rendering
- `MM.slow_tick` (slow): Combat calculations, HP/MP regen, threat table updates

### File Naming & Organization

| Type | Convention | Example |
|------|-----------|---------|
| Scripts | `snake_case.gd` | `player_combat.gd`, `spell_manager.gd` |

### Variable & Class Naming

**Classes** (use `class_name`):
```gdscript
extends Node3D

class_name SpellManager
class_name ItemResource
```

**Variables** (snake_case):
```gdscript
var jump_height: float = 2.5
var current_state: ANIMATION_STATE
var list_of_players: Array[Player]
var changes_received: Dictionary
var target_angle: float = 0.0
```

**Functions & Signals** (snake_case):
```gdscript

func transition_to(state: ANIMATION_STATE):
func send_spell(type: int, projectile_type: int, index: int):
signal died

```

**Node References** (PascalCase with `%`):
```gdscript

```

### Type Hints (Godot 4)

**Always use type hints** for parameters and return values:
```gdscript
# Correct
func add_item(item: ItemResource, amount: int) -> bool:
func teleport_to(destination: Vector3):
var tween: Tween
var data: Dictionary
var list_of_players: Array[Player] = []

# Incorrect
func add_item(item, amount):  # Missing types
```

**Godot 4 Native Types**:
- `int`, `float`, `String`, `bool`
- `Vector2`, `Vector3`, `Vector4`
- `Node`, `Node3D`, `CharacterBody3D`, `Camera3D`
- `Array[T]`, `Dictionary`, `PackedStringArray`

### RPC & Multiplayer Patterns

**RPC Decorators** (strict format - parameters must be in order: `"authority"/"any_peer"`, `"reliable"/"unreliable_ordered"`, `"call_local"/"call_remote"`):
```gdscript
# Server sends to all clients
@rpc("authority", "reliable", "call_local")
func server_state(new_position: Vector3, new_rotation: Vector3) -> void:

# Client sends to server
@rpc("any_peer", "unreliable_ordered", "call_remote")
func client_state(new_position: Vector3, new_rotation: Vector3) -> void:

# Target specific peer
@rpc("authority", "reliable", "call_local")
func teleport_to(destination: Vector3) -> void:
    teleport_to.rpc_id(id, destination)  # Send to specific player
```

**Authority Checks**:
```gdscript
if multiplayer.is_server():
    # Server-only logic (combat calculations, spawning)
if multiplayer.get_unique_id() == id:
    # Local player only (input, camera, UI)
```

### Error Handling

**1. Node Safety**:
```gdscript
if is_instance_valid(climbing_object):
    # Safe to use
if node:
    node.some_method()
```

**2. Dictionary Safety**:
```gdscript
if data.has("position"):
    global_position = data["position"]
if changes_received.has(id):
    changes_received[id].append(value)
```

**3. Cleanup**:
```gdscript
player.remove.rpc()  # RPC before queue_free
queue_free()  # Safe node removal
if tween:
    tween.kill()  # Clean up tweens before reassigning
```

### Comments & Documentation

**Minimal comments** - code should be self-documenting:
```gdscript
# Good: Only for complex logic or TODOs
var target_angle: float = 0.0  # Camera rotation offset
is_running = !is_running  # toggle run
# TODO: fix 0.15 radians off center

# Bad: Obvious comments
health = value  # Set health to value
```

**Signal Documentation**:
```gdscript
signal died
signal health_changed(change)
signal tick
signal slow_tick
```

## Component/Module Creation Guide

### Step 1: Create the Resource (Data-First)

If your feature needs data (spells, items, quests), create a `.tres` resource first:

**File**: `spellManager/spells/Projectile/new_spell_resource.tres` (or create via Editor)

**Script**: `spellManager/spells/Projectile/new_spell.gd`
```gdscript
extends Resource
class_name NewSpellResource

enum SPELLTYPE { INSTANT, PROJECTILE, CHANNEL }
enum PROJECTILE_TYPE { FIREBALL, FROSTBOLT, NEW_PROJECTILE }

@export var spell_name: String
@export var mana_cost: int
@export var cooldown: int
@export var damage: int
```

### Step 2: Create the Component Script

**File**: `components/your_feature/your_feature.gd`

```gdscript
extends Node3D
class_name YourFeature

@export var some_config: float = 1.0
@onready var parent: Player = get_parent()
@onready var some_node: Node3D = $SomeNode

signal some_event

func _ready():
    # Initialize on ready
    if multiplayer.is_server():
        MM.slow_tick.connect(handle_updates)

func _process(delta: float) -> void:
    # Per-frame logic
    pass

func some_function(param: String) -> bool:
    # Business logic
    return true

@rpc("authority", "reliable", "call_local")
func server_update(value: int):
    # Server-authoritative update
    pass
```

### Step 3: Create the Scene

**File**: `components/your_feature/your_feature.tscn`

```
[gd_scene load_steps=2 format=3 uid="uid://..."]

[node name="YourFeature" type="Node3D"]
script = ExtResource("1")

[node name="SomeNode" type="Node3D" parent="."]
```

### Step 4: Integrate into Player (if player-related)

**Edit**: `components/player/player.gd`

```gdscript
@onready var your_feature: YourFeature = $YourFeature

func _ready():
    move_and_slide()
    if multiplayer.is_server():
        your_feature.some_event.connect(handle_event)
```

### Step 5: Add Multiplayer Support (if needed)

**In your component**:
```gdscript
@rpc("any_peer", "call_remote", "reliable")
func request_action():
    if multiplayer.is_server():
        handle_action.rpc()

@rpc("authority", "reliable", "call_local")
func handle_action():
    # Process on all clients
    pass
```

### Step 6: Add to Autoload (if global)

**Edit**: `project.godot`

```ini
[autoload]

YourManager="*res://components/your_feature/manager.tscn"
```

### Step 7: Test

```bash

godot --path . --debug

# Run specific demo
godot --path . --scene res://demo/src/Demo.tscn
```

## Common Pitfalls

### 1. Bypassing the State Machine (Critical)

**Anti-Pattern:**
```gdscript
# WRONG: Directly setting velocity in _process
func _process(delta):
    parent.velocity.x = input * speed  # Breaks state architecture!
    parent.global_position.x += 5.0    # Breaks physics!
```

**Why it breaks:**
- Circumvents `GroundState`, `ClimbingState`, etc.
- Breaks multiplayer sync (server doesn't know about physics changes)
- Causes desync between states (e.g., jumping while climbing)

**Correct Pattern:**
```gdscript
# RIGHT: Use state events
state_chart.send_event("to_climbing")
# Or modify state-specific variables
parent.speed_modifier = 1.5  # States read this
```

---

### 2. Ignoring Server Authority (Critical)

**Anti-Pattern:**
```gdscript
# WRONG: Client validates combat
func _on_attack_pressed():
    if target.health > 0:  # Client-side check!
        target.health -= damage  # Direct manipulation!
        target.die()  # No server validation!
```

**Why it breaks:**
- Cheatable (clients can modify health directly)
- Desyncs with server state
- Bypasses `slow_tick` damage queuing and threat calculation

**Correct Pattern:**
```gdscript
# RIGHT: Request action from server
@rpc("any_peer", "call_remote", "reliable")
func request_damage(amount: int):
    # Server validates via changes_received dictionary
    # Then broadcasts via change_health.rpc()
    pass
```

---

### 3. Direct Node Access Without Safety Checks

**Anti-Pattern:**:
```gdscript
# WRONG: No safety checks
func _ready():
    var player = get_parent()  # Might be null!
    player.combat.died.connect(handle_death)  # Crashes if player has no CombatComponent
    
func _process(delta):
    $SomeNode.visible = true  # Crashes if node doesn't exist
```

**Why it breaks:**
- Crashes when scene hierarchy changes
- Fails for remote players (who have stripped state machines)
- Breaks when testing scenes in isolation

**Correct Pattern:**
```gdscript
# RIGHT: Safety checks everywhere
@onready var parent: Player = get_parent()
@onready var combat: CombatComponent = parent.combat

func _ready():
    if is_instance_valid(combat):
        combat.died.connect(handle_death)
    
func _process(delta):
    if $SomeNode:
        $SomeNode.visible = true
```

---

### 4. Hardcoding `load()` Instead of `preload()`

**Anti-Pattern:**
```gdscript
# WRONG: Runtime loading
var scene = load("res://path/to/scene.tscn")  # Blocks frame!
var material = load("res://material.tres")
```

**Why it breaks:**
- Runtime loading blocks the main thread
- Breaks when project path changes
- Not optimized by Godot's resource manager

**Correct Pattern:**
```gdscript
# RIGHT: Preload at top of file
const MY_SCENE = preload("res://path/to/scene.tscn")
const MY_MATERIAL = preload("res://material.tres")
```

---

### 5. Mixing Client and Server Logic

**Anti-Pattern:**
```gdscript
# WRONG: Input on server, combat on client
if Input.is_action_pressed("attack"):  # Only works on local client!
    spawn_projectile()  # Only spawns for local player!
```

**Why it breaks:**
- `Input` only works on local client (not server)
- Remote players don't see projectiles
- Server needs to validate all actions

**Correct Pattern:**
```gdscript
# Client sends input
if Input.is_action_just_pressed("attack"):
    request_attack.rpc()

# Server handles logic
@rpc("any_peer", "reliable", "call_remote")
func request_attack() -> void:
    if multiplayer.is_server():
        spawn_projectile.rpc()  # Broadcasts to all
```
